🗂️ Chapter 1.1: Introduction to Data Structures

TOPIC 1 What is a Data Structure? ⚠️

A Data Structure is a way of organizing and storing data in a computer so it can be accessed and modified efficiently.

A phone contacts list is a data structure — names and numbers stored in an organized way so you can find any contact fast.

Common Examples

Array → row of boxes, each holds one item Stack → pile of plates (Last In, First Out) Queue → people waiting in line (First In, First Out) Linked List → chain of nodes, each pointing to the next Tree → family tree (root → branches → leaves) Graph → cities connected by roads

Linear vs Non-Linear

TypeMeaningExamples
LinearElements in a sequence, one after anotherArray, Stack, Queue, Linked List
Non-LinearElements in a hierarchy or networkTree, Graph
Practice Questions

Q1. Define Data Structure. Give 3 examples.
Answer: A way of organizing data for efficient access. Examples: Array, Stack, Tree.

Q2. Is a Queue linear or non-linear? Why?
Answer: Linear — elements are arranged in sequence. First in, first out.

TOPIC 2 Algorithm vs Data Structure ⚠️
Data Structure

HOW data is stored/organized in memory.
Example: Array of phone contacts

Algorithm

The steps to process or operate on data.
Example: Binary Search to find a contact

Analogy: Data structure = the kitchen (where things are stored). Algorithm = the recipe (steps to cook). You need both.
Data StructureAlgorithm
FocusOrganization of dataSteps to solve a problem
ExampleArray, Stack, TreeBubble Sort, Binary Search
AnalogyKitchen / warehouseRecipe / instruction manual
Practice Questions

Q1. What is the difference between an algorithm and a data structure?
Answer: A data structure organizes/stores data. An algorithm is a step-by-step procedure to process it.

Q2. Give one example of an algorithm operating on a data structure.
Answer: Bubble Sort (algorithm) operating on an Array (data structure).

TOPIC 3 ADT and Abstraction ⚠️

ADT (Abstract Data Type) defines WHAT operations a data structure supports — without specifying HOW they are implemented.

Abstraction means hiding internal details and exposing only what the user needs to know.

Analogy: A TV remote is an ADT. You press "Volume Up" and it works. You don't see the electronics inside — that detail is hidden (abstraction).

Stack ADT Example

ADT: Stack push(x) → add item to top pop() → remove top item peek() → see top without removing isEmpty() → check if empty HOW it works → hidden (could be array OR linked list)
ADTData Structure
FocusWHAT (logical view)HOW (physical view)
DetailsAbstract / interfaceConcrete implementation
ExampleStack ADTArray-based Stack
⚠️ TRAP: ADT ≠ Data Structure. ADT is the interface (what it does). Data structure is the actual implementation. Common mistake: saying they are the same thing.
Practice Questions

Q1. What is an ADT? How is it different from a data structure?
Answer: ADT defines what operations exist and what they do, without implementation. A data structure is the actual concrete implementation.

Q2. Give an example of abstraction in data structures.
Answer: Calling sort() in C++ — you use it without knowing which sorting algorithm runs inside.

⚙️ Chapter 1.2: Algorithm Concepts

TOPIC 5 What is an Algorithm? ⚠️

An algorithm is a finite, step-by-step set of instructions for solving a specific problem. It takes input, processes it, and produces output.

Analogy: A recipe for injera. Exact steps, a clear start, a definite end, and always gives you injera if you follow it correctly.
TOPIC 5b 5 Properties of an Algorithm ⚠️ 🔁
⭐ This is very likely to appear in your exam. Memorize all 5 with one-line explanations.
1. Input

Has zero or more inputs.
Example: numbers to be sorted

2. Output

Has at least one output.
Example: the sorted list

3. Definiteness

Every step is clear and unambiguous.
✓ "Add 1 to x"   ✗ "Do something to x"

4. Finiteness

Must terminate after a finite number of steps.
Cannot loop forever!

5. Effectiveness

Every step must be basic and executable.
No magical or impossible steps

⚠️ TRAP

An infinite loop violates Finiteness → it is NOT a valid algorithm.
while(true){} ← invalid algorithm!

Practice Questions

Q1. List and explain the 5 properties of an algorithm.
Answer: Input, Output, Definiteness (clear steps), Finiteness (must terminate), Effectiveness (each step is doable).

Q2. Is an infinite loop a valid algorithm? Why not?
Answer: No — it violates the Finiteness property. An algorithm must always terminate in a finite number of steps.

TOPIC 6 Complexity Analysis ⚠️
Time Complexity

How the running time grows as input size n increases.

We use n, not seconds — different computers run at different speeds.

Space Complexity

How much extra memory the algorithm needs as n grows.

O(1) space = constant extra memory (a few variables only).

Complexity Classes — Fastest to Slowest

O(n²)QuadraticFast growthBubble / Selection / Insertion Sort

TOPIC 7 Big-O, Big-Ω, Big-Θ — Asymptotic Notation ⚠️ 🔁
Big-O (O) — Upper Bound

WORST CASE — algorithm takes AT MOST this long.

f(n) = O(g(n)) → f grows no faster than g Example: Linear Search → O(n) (worst: check all n items)

Analogy: "Trip takes AT MOST 2 hours."

Big-Ω (Ω) — Lower Bound

BEST CASE — algorithm takes AT LEAST this long.

f(n) = Ω(g(n)) → f grows at least as fast as g Example: Linear Search → Ω(1) (best: target is first element)

Analogy: "Trip takes AT LEAST 30 min."

Big-Θ (Θ) — Tight Bound

EXACT / ALWAYS — always exactly this long.

f(n) = Θ(g(n)) → f is BOTH O(g) AND Ω(g) Example: Bubble Sort → Θ(n²) (always exactly O(n²) and Ω(n²))

Analogy: "Trip always takes exactly 1h ± 10min."

NotationBound TypeMeaningCase
O(g(n))UpperAt MOST g(n)Worst Case
Ω(g(n))LowerAt LEAST g(n)Best Case
Θ(g(n))TightEXACTLY g(n)Average / Always
Practice Questions

Q1. What is Big-O notation? What does O(n²) mean?
Answer: Big-O gives the upper bound (worst case) growth rate. O(n²) means time grows proportionally to n² as input size n increases.

Q2. What is the difference between O, Ω, and Θ?
Answer: O = upper bound (worst), Ω = lower bound (best), Θ = tight bound (exact/always).

TOPIC 8 Best, Average, Worst Case ⚠️

For the same algorithm, different inputs give different running times.

Best Case → luckiest input possible (fastest result) Average Case → typical / random input Worst Case → unluckiest input (slowest result)
CaseMeaningLinear SearchBubble Sort
Best Luckiest input Target at index 0 → O(1) Already sorted → O(n)
Average Typical random input Target near middle → O(n) Random order → O(n²)
Worst Unluckiest input Target last or not found → O(n) Reverse sorted → O(n²)
We usually focus on worst case — it guarantees the maximum time the algorithm will EVER take.
Practice Questions

Q1. For linear search, what is the best and worst case time complexity?
Answer: Best case: O(1) (target is first element). Worst case: O(n) (target is last or not found).

Q2. Why do we usually focus on worst-case complexity?
Answer: It gives an upper limit guarantee — the maximum time the algorithm will ever take, which is most useful for reliability.

🔀 Chapter 2: Sorting Algorithms

Sort 1  Bubble Sort ⚠️ 🔁

Principle: Compare adjacent elements. Swap if in wrong order. After each pass, the largest element "bubbles up" to the end. Repeat until sorted.

Analogy: Sorting playing cards — compare two side-by-side, swap if needed, keep going pass by pass.

Algorithm

  1. Outer loop: i = 0 to n-2 (number of passes)
  2. Inner loop: j = 0 to n-2-i (compare adjacent pairs)
  3. If arr[j] > arr[j+1]: swap them
  4. Repeat until no swaps needed
For i = 0 to n-2: For j = 0 to n-2-i: If arr[j] > arr[j+1]: swap(arr[j], arr[j+1])
Full Worked Example — Sort [5, 3, 8, 1]
Pass 1: [5, 3, 8, 1] 5>3 → swap → [3, 5, 8, 1] [3, 5, 8, 1] 5<8 → keep → [3, 5, 8, 1] [3, 5, 8, 1] 8>1 → swap → [3, 5, 1, 8] ← 8 settled at end Pass 2: [3, 5, 1, 8] 3<5 → keep [3, 5, 1, 8] 5>1 → swap → [3, 1, 5, 8] ← 5 settled Pass 3: [3, 1, 5, 8] 3>1 → swap → [1, 3, 5, 8] ✓ SORTED
Best Case — O(n)

Array already sorted → no swaps. With early-exit optimization.

Average Case — O(n²)

Random order input.

Worst Case — O(n²)

Reverse sorted array. Maximum swaps every pass.

Practice Questions

Q1. Sort [4, 2, 7, 1] using Bubble Sort. Show Pass 1 completely.
Answer: 4>2→swap→[2,4,7,1] | 4<7→keep | 7>1→swap→[2,4,1,7]

Q2. What is the worst-case time complexity of Bubble Sort? When does it occur?
Answer: O(n²). When the array is sorted in reverse order — maximum swaps are needed at every pass.

Sort 2  Selection Sort ⚠️ 🔁

Principle: Find the minimum element in the unsorted portion. Swap it to the front of that portion. Repeat for the rest.

Analogy: Sorting books — scan the pile, pick the smallest-numbered book, place it first. Scan the rest, pick the next smallest, place it second. Keep going.

Algorithm

  1. Set minIndex = i (assume current position is min)
  2. Scan from i+1 to end — find actual minimum position
  3. Swap arr[i] with arr[minIndex]
  4. Move i forward and repeat
For i = 0 to n-2: minIndex = i For j = i+1 to n-1: If arr[j] < arr[minIndex]: minIndex = j swap(arr[i], arr[minIndex])
Full Worked Example — Sort [5, 3, 8, 1]
Pass 1: scan [5,3,8,1] → min=1 at index 3 → swap index 0 and 3 → [1, 3, 8, 5] Pass 2: scan [3,8,5] → min=3 at index 1 → already in place → [1, 3, 8, 5] Pass 3: scan [8,5] → min=5 at index 3 → swap index 2 and 3 → [1, 3, 5, 8] ✓ SORTED
⭐ KEY FACT — Always O(n²): Selection Sort is ALWAYS O(n²) regardless of input order. It always scans the entire unsorted portion to find the minimum — it never stops early.
CaseTimeWhy?
BestO(n²)Still scans the entire unsorted part
AverageO(n²)No early exit possible
WorstO(n²)Never improves
Practice Questions

Q1. Sort [6, 2, 4, 1] using Selection Sort. Show all passes.
Answer: Pass1:min=1→[1,2,4,6] | Pass2:min=2 in place | Pass3:min=4 in place → [1,2,4,6] ✓

Q2. Why is Selection Sort always O(n²) even in the best case?
Answer: Because it always scans the entire unsorted portion to find the minimum — it never stops early, regardless of the input order.

Sort 3  Insertion Sort ⚠️ 🔁

Principle: Take one element at a time. Insert it into the correct position among the already-sorted elements on the left, shifting others right to make room.

Analogy: Sorting cards in your hand. Pick up one card at a time and slide it into the right spot among the cards you're already holding.

Algorithm

  1. Start from index 1 (first element is trivially sorted)
  2. Save arr[i] as key
  3. Compare key with elements to its left, shift right if larger
  4. Insert key at correct position
For i = 1 to n-1: key = arr[i] j = i - 1 While j >= 0 AND arr[j] > key: arr[j+1] = arr[j] ← shift right to make room j = j - 1 arr[j+1] = key ← insert key in correct spot
Full Worked Example — Sort [5, 3, 8, 1]
Step 1: key=3. arr[0]=5 > 3 → shift right. Insert 3 at front. → [3, 5, 8, 1] Step 2: key=8. arr[1]=5 < 8 → stop. 8 stays in place. → [3, 5, 8, 1] Step 3: key=1. arr[2]=8>1→shift, arr[1]=5>1→shift, arr[0]=3>1→shift. Insert 1 at front. → [1, 3, 5, 8] ✓ SORTED
Best Case — O(n)

Already sorted → no shifts needed, one comparison per element.

Average Case — O(n²)

Random order → partial shifts needed.

Worst Case — O(n²)

Reverse sorted → maximum shifts at every step.

Practice Questions

Q1. Sort [4, 2, 5, 1] using Insertion Sort. Show all steps.
Answer: key=2: 4>2 shift → [2,4,5,1] | key=5: 4<5 stay → [2,4,5,1] | key=1: shift 5,4,2 → [1,2,4,5] ✓

Q2. When is Insertion Sort O(n)?
Answer: When the array is already sorted — no shifts are needed, just one comparison per element in the outer loop.

🔍 Chapter 2: Searching Algorithms

Search 1  Linear Search ⚠️

Principle: Check each element one by one from start to end until you find the target or exhaust the array.

Analogy: Scrolling through an unsorted contact list looking for a name — you go one by one from top to bottom.

Algorithm

For i = 0 to n-1: If arr[i] == target: return i ← FOUND at index i return -1 ← NOT FOUND
Worked Example — Search for 8 in [3, 7, 8, 2, 5]
Check index 0: arr[0] = 3 ≠ 8 → continue Check index 1: arr[1] = 7 ≠ 8 → continue Check index 2: arr[2] = 8 = 8 → FOUND at index 2 ✓
CaseTimeCondition
BestO(1)Target is the very first element
AverageO(n)Target is near the middle
WorstO(n)Target is last element OR not in array
Linear search works on both sorted and unsorted arrays. No requirements.
Search 2 ⭐  Binary Search ⚠️ (MOST IMPORTANT)

Principle: Only works on a sorted array. Compare target with the middle element. Eliminate half the search space each time.

Analogy: Dictionary search — open the middle, decide left or right, keep halving until you find the word. Much faster than reading every page.

Algorithm

  1. Set low = 0, high = n-1
  2. Compute mid = (low + high) / 2
  3. If arr[mid] == target → FOUND, return mid
  4. If arr[mid] < target → target is in right half, set low = mid + 1
  5. If arr[mid] > target → target is in left half, set high = mid - 1
  6. Repeat from step 2. If low > high → NOT FOUND, return -1
low = 0, high = n-1 While low <= high: mid = (low + high) / 2 If arr[mid] == target: return mid ← FOUND Else if arr[mid] < target: low = mid + 1 ← search RIGHT half Else: high = mid - 1 ← search LEFT half return -1 ← NOT FOUND
Full Worked Example — Search for 7 in [1, 3, 5, 7, 9, 11, 13]
Array: [ 1, 3, 5, 7, 9, 11, 13] Index: 0 1 2 3 4 5 6 Step 1: low=0, high=6 mid = (0+6)/2 = 3 arr[3] = 7 == 7 → FOUND at index 3 ✓ (1 step!) ---- Search for 11: Step 1: mid=3, arr[3]=7 < 11 → low=4 Step 2: mid=(4+6)/2=5, arr[5]=11 == 11 → FOUND ✓ Search for 6 (not in array): Step 1: mid=3, arr[3]=7 > 6 → high=2 Step 2: mid=1, arr[1]=3 < 6 → low=2 Step 3: mid=2, arr[2]=5 < 6 → low=3 Step 4: low=3 > high=2 → return -1 (NOT FOUND)
CaseTime
BestO(1)
AverageO(log n)
WorstO(log n)

With 1024 elements → only 10 steps needed (log₂1024 = 10).

⚠️ CRITICAL TRAP

Binary Search REQUIRES a sorted array. On an unsorted array, the elimination logic is wrong and you will get incorrect results. Always verify the array is sorted first.

Practice Questions

Q1. Search for 5 in sorted array [2, 4, 5, 8, 10, 15] using Binary Search. Show all steps.
Answer: low=0, high=5, mid=2, arr[2]=5 = target → FOUND at index 2 ✓ (1 step)

Q2. Why can't Binary Search work on an unsorted array?
Answer: Binary Search eliminates half the array based on comparison with middle. This only works if elements are in order — otherwise the direction decision is wrong.

BONUS Pointer Sort — Basic Idea

Instead of moving actual data (which can be large/expensive to move), sort an array of pointers (addresses) that point to the data. Only the pointers move, not the data itself.

Analogy: Instead of rearranging heavy boxes in a warehouse, you rearrange sticky notes (labels) that say which box is where. The boxes don't move — only the notes change order.
Data: ["Zara", "Abel", "Mike"] ← stays in place Pointers: [ 0, 1, 2 ] ← original order After pointer sort (alphabetical): Pointers: [ 1, 2, 0 ] ← Abel, Mike, Zara Data: unchanged ✓

When to use: When data records are very large — moving pointers is much faster than swapping large objects.

  • Ω(g(n)) — Lower bound, BEST case, "at least g(n)"

Idea: Swap adjacent pairs, bubble max to end each pass

SELECTION SORT

Idea: Find minimum in unsorted part, place at front

INSERTION SORT

Idea: Insert each element into correct position on left side

Best: O(n) (sorted) | Average: O(n²) | Worst: O(n²) (reverse) | Space: O(1)

LINEAR SEARCH

Idea: Check elements one by one until found or end

Best: O(1) | Worst: O(n) | Works on unsorted arrays

BINARY SEARCH ⭐ (Most Important)

Idea: Compare target with middle, eliminate half. mid = (low + high) / 2

Best: O(1) | Worst: O(log n) | REQUIRES sorted array!

  2. Output — At least one output. Example: the sorted list.

Question 2 · Easy-Medium · Bubble Sort
Sort the array [6, 3, 9, 2] using Bubble Sort.
Show every comparison and swap in all passes. State the time complexity.
Solution:
Pass 1: 6>3 → swap → [3, 6, 9, 2] 6<9 → keep → [3, 6, 9, 2] 9>2 → swap → [3, 6, 2, 9] ← 9 settled Pass 2: 3<6 → keep → [3, 6, 2, 9] 6>2 → swap → [3, 2, 6, 9] ← 6 settled Pass 3: 3>2 → swap → [2, 3, 6, 9] ✓ SORTED

Time Complexity: O(n²) — worst/average case. Best case O(n) if already sorted.

Question 3 · Medium · Big-O, Ω, Θ
(a) What is the difference between Big-O, Big-Ω, and Big-Θ notation?
(b) For Selection Sort, state the Big-O, Big-Ω, and Big-Θ. Justify your answer.
Solution:

(a)

  • O(g(n)) — Upper bound (worst case). Algorithm takes AT MOST g(n) time.
  • Ω(g(n)) — Lower bound (best case). Algorithm takes AT LEAST g(n) time.
  • Θ(g(n)) — Tight bound (always). Algorithm always takes exactly g(n) time (within constants).

(b) Selection Sort:

  • O(n²) — Worst case is O(n²)
  • Ω(n²) — Even the best case is O(n²) because it always scans the entire unsorted portion
  • Θ(n²) — Since best and worst are both n², the tight bound is Θ(n²)

Selection Sort is Θ(n²) — it always takes exactly n² time.

Question 4 · Medium · Binary Search
Search for target = 14 in the sorted array [2, 5, 8, 11, 14, 17, 23] using Binary Search.
Show every step (low, high, mid, comparison). State the time complexity.
Solution:
Array: [ 2, 5, 8, 11, 14, 17, 23] Index: 0 1 2 3 4 5 6 Step 1: low=0, high=6 mid = (0+6)/2 = 3 arr[3] = 11 < 14 → go RIGHT, low = 4 Step 2: low=4, high=6 mid = (4+6)/2 = 5 arr[5] = 17 > 14 → go LEFT, high = 4 Step 3: low=4, high=4 mid = (4+4)/2 = 4 arr[4] = 14 == 14 → FOUND at index 4 ✓

Found in 3 steps. Time Complexity: O(log n) = O(log 7) ≈ 3 steps.

Question 5 · Hard · Compare + Explain
(a) Sort [7, 2, 9, 1, 5] using Insertion Sort. Show all steps.
